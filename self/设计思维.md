# 设计思维沉淀

## 1. 用“配置 + Factory + Registry + Refresh + 代理入口”管理可切换对象

这个项目里模型配置的设计很有借鉴价值。它没有把某一个具体模型直接写死成固定 Bean，而是把“容易变化的东西”先抽到配置里，比如 `baseUrl`、`apiKey`、`modelName`、请求路径、代理参数等。业务代码不直接关心这些细节，而是统一依赖 `ChatModel`、`EmbeddingModel` 这类抽象接口。

背后的核心思路是：**固定的是入口，变化的是入口背后的真实实现**。

在这个项目里，这条链路大概是这样的：

```text
数据库中的模型配置
    -> DynamicModelFactory 根据配置创建真实模型对象
    -> AiModelRegistry 缓存当前可用模型
    -> refreshChat() / refreshEmbedding() 让旧缓存失效
    -> DataAgentConfiguration 暴露稳定的代理 Bean 给业务层使用
```

这套设计不是“每次调用都重新创建对象”。真正的做法是：第一次用到时按当前配置创建对象，然后放进缓存里复用；当配置发生变化时，通过 refresh 方法把缓存清空，下一次调用再按最新配置重新创建。也就是说，它的“动态”不在于每次都变，而在于**运行时可以根据配置决定创建谁，并且可以在刷新后切换到新的实现**。

这种思路适合用在“对象本身比较重、但背后配置可能变化”的地方。比如模型客户端、支付渠道客户端、短信渠道客户端、第三方 API 客户端、文件存储客户端等。它们都有一个共同特点：业务层只想调用一个稳定接口，但底层具体用哪个厂商、哪个地址、哪个密钥、哪个模型，最好不要写死在业务代码里。

以后如果让 AI 按这个思路实现，可以直接给它这个模式：

```text
请把可变配置抽成配置对象或数据库配置；
请写一个 Factory，把配置翻译成真实可调用对象；
请写一个 Registry，负责缓存当前对象、懒加载、并发安全和刷新；
请让业务层只依赖抽象接口，不直接依赖具体实现；
如果需要热切换，请提供 refresh 方法清空缓存，让下一次调用重新按最新配置创建。
```

一句话总结：
> 不要让业务代码直接依赖某个固定实现，而是让业务依赖稳定入口；具体实现由配置决定，由 Factory 创建，由 Registry 管理，由 Refresh 触发切换。
