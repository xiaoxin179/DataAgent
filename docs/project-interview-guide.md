# DataAgent 项目面试题

## 一、正确的面试表达方式

这类项目在真实面试里，面试官**不会先知道你的源码类名**，也不会围绕 `EvidenceRecallNode`、`SchemaRecallNode` 这种代码实现名来问。

真实面试顺序通常是：

1. 你先介绍项目背景、目标、模块、你的职责
2. 面试官基于你的介绍追问
3. 追问重点通常是：
   - 这个项目整体怎么分模块
   - 你负责了哪些功能
   - 某个核心功能你怎么实现的
   - 为什么这么设计，而不是别的方案
   - 遇到过什么难点、问题、线上风险
   - 你做了哪些优化

所以这份文档会按照**真实面试场景**来写，不从源码类名出发，而从“你对外怎么讲项目”出发。

---

## 二、这份文档怎么用更合理

这份文档现在的内容已经够覆盖一次比较深入的项目面试，但如果直接从头背到尾，会有两个问题：

1. 信息量太大，容易失去重点
2. 面试真实推进顺序，不会完全按“题库顺序”来

所以更推荐你按下面这个顺序使用：

### 1. 第一层，只背 4 个高优先级内容

- 项目一句话介绍
- 项目模块划分
- 你负责了哪些功能
- 这个项目为什么不是简单 NL2SQL，而是 Agent 系统

### 2. 第二层，重点准备 3 条主线

- 数据理解主线：怎么把业务问题映射成数据结构
- 分析执行主线：怎么从计划走到 SQL / Python / 报告
- Agent 主线：为什么这是 Agent，怎么做到可规划、可执行、可纠错、可恢复

### 3. 第三层，再补难点和优化

- 难点
- 风险
- 性能优化
- 后续演进方向

### 4. 最像真实面试的回答顺序

如果面试官先让你介绍项目，你最合理的展开顺序是：

1. 项目目标
2. 模块划分
3. 你的职责
4. 核心链路
5. 关键设计取舍
6. Agent / RAG
7. 难点、优化、风险

也就是说，这份文档虽然是题库，但你真正使用时，应该按“项目介绍 -> 模块职责 -> 核心能力 -> Agent 设计 -> 难点优化”这个顺序来准备。

## 三、你可以这样介绍项目

### 1. 项目一句话介绍

这个项目是一个企业级数据分析 Agent，目标是把用户的自然语言问题，转成可执行的数据分析流程，最终输出 SQL 查询结果、深度分析结果和总结报告。

### 2. 项目解决的问题

传统 BI 或报表系统有几个问题：

- 业务人员不会写 SQL
- 数据库表多、字段多，理解门槛高
- 单纯做 NL2SQL 很容易生成错 SQL
- 有些分析不能只靠 SQL，还需要二次计算和统计分析

所以这个项目不是简单做一个“问一句，吐一条 SQL”，而是做了一整套数据分析链路。

### 3. 项目模块划分

这个项目我会对外拆成 6 个模块来讲：

1. **请求接入与会话管理模块**
   - 接收用户问题
   - 管理一次分析请求的上下文和线程标识

2. **问题理解与知识增强模块**
   - 识别用户问题是不是数据分析请求
   - 结合业务知识、术语知识增强用户问题

3. **数据理解模块**
   - 从候选数据源、候选表、候选字段里筛出和问题最相关的结构
   - 解决表多、字段多、关联复杂的问题

4. **分析规划与执行模块**
   - 先生成分析计划
   - 再按计划决定是走 SQL 查询，还是继续做 Python 深度分析

5. **结果生成模块**
   - 汇总 SQL 结果、分析结果
   - 最终生成结构化分析报告

6. **模型与执行基础设施模块**
   - 动态切换大模型
   - 管理向量检索
   - 管理 Python 执行环境
   - 管理多数据源连接

### 4. 你的职责建议这样讲

你可以按“主链路 + 难点能力”来讲，而不是说“我写了几个类”。

建议口径：

> 我主要负责的是后端主链路设计和核心能力落地，重点包括四块：  
> 第一是数据分析工作流编排，把用户问题到结果输出拆成多阶段链路；  
> 第二是数据理解能力，包括知识增强、候选表字段筛选、表关系补全；  
> 第三是 SQL 生成后的校验和纠错机制，避免生成能跑但不正确的 SQL；  
> 第四是 Python 深度分析执行链路，包括代码生成、隔离执行、失败重试和降级。

这个说法更像高级面试，不像“我写了几个接口”。

## 1. 项目整体类

### Q1：你先整体介绍一下这个项目。

**A：**

这个项目是一个企业级数据分析 Agent。用户输入自然语言问题之后，系统不会直接一步生成 SQL，而是先做问题理解、业务知识增强、数据结构筛选，再生成分析计划，最后再按计划去执行 SQL 或 Python 分析，最终输出结果和报告。  
它本质上是在做“自然语言数据分析流程自动化”，不是单点的 NL2SQL。

### Q2：这个项目和普通的 NL2SQL 项目有什么区别？

**A：**

普通 NL2SQL 项目很多是“问题进来，prompt 一下，直接出 SQL”。  
这个项目的区别在于，它把整件事拆成多个阶段：

- 先判断用户问的是不是分析问题
- 再补充业务知识和术语理解
- 再筛选最相关的数据表和字段
- 再生成计划
- 再做 SQL 执行、纠错、深度分析和报告生成

所以它是一个完整的分析 Agent，而不是一个 SQL 生成器。

### Q3：这个项目有哪些核心模块？

**A：**

我会拆成 6 个模块：

1. 请求接入与会话管理
2. 问题理解与知识增强
3. 数据理解，也就是候选表、字段、关系筛选
4. 分析规划与执行
5. 结果生成与报告输出
6. 模型、检索、执行器这些基础设施模块

如果面试官继续追问，我会再重点展开我负责的主链路模块。

### Q4：你在这个项目里主要负责什么？

**A：**

我主要负责后端主链路和核心能力设计，重点是四块：

1. 把用户问题到结果输出拆成一条可编排的分析链路
2. 做数据理解能力，包括知识增强、候选表字段筛选和表关系补全
3. 做 SQL 生成后的校验和纠错回路
4. 做 Python 深度分析执行链路，包括隔离执行、失败重试和降级

如果面试官希望更细，我会再拆成“流程设计、核心模块实现、稳定性保障、性能优化”四个维度讲。

## 2. 模块职责与核心设计类

### Q5：为什么你们没有把它做成“用户提问直接生成 SQL”，而是拆成这么多阶段？

**A：**

因为真实业务数据场景里，直接一步生成 SQL 风险太高。  
主要问题有三个：

1. 用户问题本身表达不规范，业务术语很多
2. 数据库表多、字段多，模型很容易选错
3. 即使 SQL 能执行，也不代表答案正确

所以我把它拆成多阶段，本质上是在做“逐步收敛”，先降低歧义，再降低选表错误，再降低 SQL 错误，最后再做结果增强。

### Q6：为什么要先做“问题理解”和“知识增强”？

**A：**

因为业务用户提问往往不是数据库语言，而是业务语言。  
比如他说“核心用户”“高价值客户”“爆款商品”，这些词在数据库里不一定直接对应表名字段名。  
如果不先做知识增强，后面选表、选字段和生成 SQL 都容易偏。  
所以我把这一步放在前面，先把用户问题变成更适合后续分析链路处理的表达。

### Q7：为什么要单独做“数据理解模块”？

**A：**

因为数据分析问题的难点之一，不是 SQL 语法，而是“到底该查哪张表、哪些字段、怎么关联”。  
很多数据库里表非常多，如果直接把全库结构丢给模型，噪声太大，准确率反而下降。  
所以我单独做了数据理解模块，先缩小候选范围，再把更精炼的数据结构信息交给后续分析和 SQL 生成。

### Q8：为什么要先生成计划，再执行？

**A：**

因为复杂分析问题不是一步能完成的。  
有些问题可能先查基础数据，再做进一步分析，再输出总结。  
如果不先做计划，系统每一步都容易局部最优，看不到全局目标。  
所以我先让系统生成分析计划，再按计划决定每一步是查数、分析，还是总结。

## 3. 核心功能实现类

### Q9：你说你负责了“数据理解能力”，具体是怎么实现的？

**A：**

我的思路是分层筛选，而不是一步把所有数据库结构交给模型。

实现上我大概分三步：

1. 先根据增强后的问题，从候选范围里筛出最相关的表
2. 再基于这些表补充字段信息
3. 最后再补表和表之间的关系信息，形成更完整的结构化上下文

这么做的目的，是把全库噪声降下来，让后续 SQL 生成只围绕最相关的数据结构展开。

### Q10：你刚才说“补表关系”，这个为什么重要？

**A：**

因为真实业务库里，经常没有很规范的物理外键，但业务上表和表之间其实有关系。  
如果系统只依赖数据库原生外键，很多 JOIN 是推不出来的。  
所以我在设计里加入了“逻辑关系补全”的能力，用业务维护的关系去补数据库缺失的关联信息。  
这对提升 SQL 准确率非常关键。

### Q11：SQL 生成这块你是怎么实现的？

**A：**

我不是做成“一次生成完就直接执行”，而是做成了一个可纠错的闭环。  
流程是：

1. 先基于问题、知识增强结果、数据结构信息生成 SQL
2. 生成后不立刻执行，先做一层语义校验
3. 如果语义不通过，回到 SQL 生成阶段修正
4. 如果语义通过，再执行 SQL
5. 如果 SQL 执行报错，再把错误信息反馈回生成阶段，继续修复

这个设计能明显提高真实可用性。

### Q12：为什么 SQL 生成后还要做语义校验？

**A：**

因为 SQL 语法正确不代表结果正确。  
真实问题里最常见的不是“SQL 跑不起来”，而是“SQL 跑起来了，但查错了”。  
比如：

- 时间范围理解错
- 聚合口径错
- 表选错
- 过滤条件缺失

所以我单独加了一层语义校验，让系统在执行前先判断这条 SQL 是否真的在回答用户问题。

### Q13：SQL 执行报错以后，你是怎么做纠错的？

**A：**

我会把执行错误信息反馈回 SQL 生成阶段，让系统基于“原 SQL + 报错原因”重新生成。  
这本质上是一个失败驱动的自修复机制。  
当然我也会限制最大重试次数，避免系统陷入无限回路。

### Q14：你负责的 Python 深度分析链路是怎么实现的？

**A：**

我把它拆成三步：

1. 先根据 SQL 结果和当前分析目标生成 Python 代码
2. 再在受控执行环境中跑这段代码
3. 最后把执行结果转成用户可读的分析输出

这样 SQL 负责取数，Python 负责复杂分析，职责比较清晰。

### Q15：为什么要加 Python 分析，而不是全都交给 SQL？

**A：**

因为 SQL 更擅长结构化查询和聚合，但不擅长复杂的二次统计、规则计算、数据变换和部分算法分析。  
当分析需求超过 SQL 的舒适区时，我会把它交给 Python。  
这样系统能力更完整，不会被 SQL 表达能力卡死。

### Q16：Python 执行环境你为什么要做隔离？

**A：**

因为 Python 代码是模型生成的，不是人工写死的，天然有不确定性。  
如果直接在宿主环境执行，风险很大。  
所以我在设计里会使用隔离执行环境，并加上内存、CPU、超时等限制，尽量保证即使代码出问题，也不会影响主系统稳定性。

### Q17：如果 Python 执行失败了怎么办？

**A：**

我会先让系统基于失败信息重试修复。  
如果达到最大重试次数还失败，我不会让整条链路直接崩掉，而是做降级，让流程继续返回一个受限结果。  
因为对业务系统来说，稳定性通常优先于极限准确率。

## 4. 架构取舍类

### Q18：你为什么要把主链路设计成工作流，而不是写成一个大 Service？

**A：**

因为这个项目不是固定线性流程，它有很多条件分支和回路：

- 有些问题会提前结束
- 有些问题会走 SQL
- 有些问题会走 Python
- SQL 可能重试
- 人工审核还会中断和恢复

如果写成大 Service，后面会充满大量 if/else，扩展和维护都很差。  
工作流方式更适合这种复杂、非确定性的分析链路。

### Q19：为什么你要做“分阶段收敛”，而不是做一个超强 prompt？

**A：**

因为 prompt 再强，也很难同时解决业务语义、数据结构筛选、SQL 正确性、执行错误、复杂分析这几类问题。  
这不是一个“提示词不够强”的问题，而是系统工程问题。  
所以我更倾向于把复杂问题拆成多个子问题，每个阶段解决一部分不确定性。

### Q20：为什么你选择让系统先筛表、再筛字段、再补关系？

**A：**

因为这是最符合真实数据库噪声控制的路径。  
如果直接全库级别筛字段，干扰项太多；  
如果先缩小表范围，再看字段，再补关系，信息量更可控，模型也更容易稳定。

### Q21：为什么你要做人工审核能力？

**A：**

因为企业数据分析场景里，有些查询和分析是高风险的，不能完全放给 AI 自动执行。  
尤其当系统已经生成了计划，但用户希望确认“系统理解得对不对”的时候，人工审核可以让链路从“全自动”切换成“半自动可控”。  
这类能力在 ToB 场景里很重要。

### Q22：为什么不直接把人工审核做成普通确认环节，而要放进主链路里？

**A：**

因为如果只是前端弹窗确认，本质上只是一个页面交互，没有真正进入系统执行状态。  
我把它放进主链路里，是为了让“暂停、修改、恢复”都基于同一条分析上下文来做，而不是用户确认后又重新跑一遍。  
这样状态更一致，也更适合企业级流程控制。

## 5. 难点与问题类

### Q23：这个项目你觉得最难的点是什么？

**A：**

我觉得最难的是三件事：

1. 怎么让系统选对表、选对字段、理解对数据关系
2. 怎么让 SQL 不只是能执行，而是真的能回答问题
3. 怎么让 Python 这类高风险能力可控地接进主链路

这三个点分别对应的是数据理解、语义校验和执行隔离。

### Q24：你在这个项目里遇到过什么典型问题？

**A：**

比较典型的问题有：

- 用户问题业务术语太重，数据库字段根本不直接对应
- 数据库缺失外键信息，系统很难自动生成正确 JOIN
- SQL 有时候能执行，但结果口径不对
- Python 代码生成后执行不稳定

这些问题都不是单一技术能解决的，所以我才会做成多阶段链路。

### Q25：你怎么解决“能跑但不准”的问题？

**A：**

我主要做了三层：

1. 前置知识增强，减少问题理解偏差
2. 数据结构筛选，减少选错表和字段
3. SQL 生成后的语义校验和纠错回路

也就是说，我不是只盯着执行层，而是从输入、理解、生成、执行四个阶段一起控风险。

### Q26：如果逻辑关系配错了，会有什么后果？

**A：**

这个后果很严重。  
因为一旦逻辑关系错了，后面 JOIN 的方向和条件可能系统性错误，而且 SQL 还不一定报错。  
所以这类问题比“直接执行失败”更危险，因为它会产生看起来正常、其实错误的结果。

## 6. 性能优化与稳定性类

### Q27：你在这个项目里做了哪些性能优化？

**A：**

我主要做了几类：

1. 用流式返回，降低用户等待感知
2. 对模型实例做缓存，避免每次重新初始化
3. 把检索过程做并行化
4. 控制大模型输入大小，比如只传样本数据，不传全量结果
5. 对重试次数做上限控制，避免无效资源消耗

### Q28：你是怎么控制大模型调用成本的？

**A：**

我会重点控制两件事：

1. 控制输入上下文大小，比如先筛选数据结构，再给模型
2. 结果样本化，比如 Python 和图表生成都只传样本结果，不传全量数据

因为这个项目最大的成本来源之一，不是单次调用次数，而是每次上下文太大。

### Q29：你是怎么保证系统稳定性的？

**A：**

我会从三层保证：

1. 流程层：分阶段执行，关键节点可校验、可重试、可中断
2. 执行层：Python 隔离运行，限制资源
3. 容错层：重试上限、失败回退、降级输出

也就是说，稳定性不是某一个点做出来的，而是整条链路的设计结果。

## 7. 后续演进类

### Q30：如果让你继续优化这个项目，你会先做什么？

**A：**

我会先做三件事：

1. 把运行态从单机内存迁移到可分布式的状态存储
2. 继续增强 SQL 执行安全，比如 AST 校验和只读权限控制
3. 进一步做计划和上下文压缩，控制大模型调用成本

这样会同时提升系统的可扩展性、安全性和成本控制能力。

### Q31：这个项目目前最大的架构风险是什么？

**A：**

最大的风险是运行态比较依赖单机内存。  
如果后续上多实例，会涉及：

- 分析链路上下文共享
- 中断恢复状态同步
- 长连接会话路由

所以如果要上更大规模，第一优先级不是加功能，而是先解决状态分布式问题。

### Q32：如果以后要支持更多分析工具，这套架构还能撑住吗？

**A：**

可以。  
因为现在的设计本身就是工作流化的，新增一个分析工具，本质上是：

- 扩一个执行能力
- 扩一个计划节点类型
- 扩一个调度规则

整体架构不需要推翻，只需要沿着现有抽象扩展。

## 8. Agent / RAG 认知与业务理解类

### Q33：你为什么把这个项目定义成 Agent，而不是一个普通的数据问答系统？

**A：**

因为它不是“收到问题 -> 返回答案”的单轮问答系统，而是具备了更完整的任务分解和执行能力。  
它会先理解问题，再结合知识和数据结构做分析规划，然后决定走 SQL 查询、深度分析还是报告生成。  
也就是说，它不是只会回答，而是会围绕目标做多步骤决策和执行，所以我把它定义成 Agent，而不是普通聊天问答。

### Q34：那这个项目里的 Agent 具体体现在哪些地方？

**A：**

我觉得主要体现在四点：

1. 它会围绕用户目标生成分析计划，而不是直接吐结果
2. 它会根据不同步骤选择不同执行能力，比如 SQL 或 Python
3. 它有自我纠错能力，比如 SQL 失败后会回退修复
4. 它支持人工审核和恢复执行，说明它不是一次性调用，而是一个可控任务体

所以它更像一个“会规划、会执行、会纠错”的分析智能体。

### Q35：如果面试官质疑你这个项目“本质上还是工作流，不算 Agent”，你怎么答？

**A：**

我会先承认它确实用了工作流骨架，但关键不在于“有没有工作流”，而在于“中间决策是不是动态产生的”。  
传统工作流更多是把步骤和路径提前写死，而这个项目里很多关键决策是运行时由模型结合上下文做出来的，比如：

- 用户问题怎么理解
- 应该查哪些表
- 当前更适合走 SQL 还是 Python
- SQL 出错后怎么修复

也就是说，这个系统不是“纯工作流”，而是“工作流骨架 + Agent 决策内核”。  
如果没有后面这层动态理解、规划、工具选择和纠错，它才只是一个普通流程编排系统。

### Q36：这个项目里的 RAG 主要解决什么问题？

**A：**

RAG 在这里最主要解决的是“业务语言和数据库语言不一致”的问题。  
真实业务用户问问题时，常常说的是业务术语，不是表名和字段名。  
比如他说“高价值客户”“核心用户”“爆款商品”，数据库里不一定就叫这个名字。  
如果不先把这些业务知识补进来，后面的选表、选字段、生成 SQL 都容易偏。

### Q37：为什么这个项目不能只靠数据库 Schema，不做 RAG？

**A：**

因为 Schema 只能告诉你数据库结构，但不能完整表达业务语义。  
数据库里可能有：

- 表名和业务概念不一致
- 字段名很技术化，用户根本不会这么说
- 一些口径定义根本不在库结构里，而在业务文档、FAQ、术语表里

所以只靠 Schema，系统只知道“有什么数据”，但不一定知道“业务上这是什么意思”。  
RAG 的作用就是把这部分业务语义补齐。

### Q38：你这个项目里的 RAG 都召回什么内容？

**A：**

我对外一般会讲成两类：

1. **业务知识**
   - 比如术语定义、指标口径、业务规则

2. **智能体私有知识**
   - 比如 FAQ、问答知识、业务文档片段、经验知识

这两类知识结合起来，能让系统更好理解“用户到底在问什么”和“应该按什么业务口径来分析”。

### Q39：你们为什么不把 RAG 直接放到 SQL 生成那一步，而是提前放到链路前面？

**A：**

因为 RAG 的价值不只是帮助生成 SQL，它更早地影响了整个链路：

- 它会影响问题理解
- 会影响候选表字段筛选
- 会影响分析计划生成
- 最后才会影响 SQL

如果只在 SQL 生成那一步才注入知识，那前面的理解和规划阶段已经可能偏掉了。  
所以我更倾向于把 RAG 前置，先补语义，再走后续流程。

### Q40：你的 RAG 为什么要先做 query rewrite，再去检索？

**A：**

因为用户原始提问经常不是一个适合检索的表达。  
它可能依赖上下文、语义省略很多，甚至带口语化表达。  
我先把用户问题改写成更独立、更完整的检索语句，再去查知识库，召回质量会更稳定。  
这一步本质上是在优化“检索输入质量”。

### Q41：你怎么理解这个项目里 Agent 和 RAG 的关系？

**A：**

我会把它们理解成“决策大脑”和“外部知识补给”的关系。  
Agent 负责规划、决策、执行；RAG 负责给 Agent 提供它原本不知道、但当前问题又必须依赖的业务知识。  
如果没有 Agent，RAG 只能做检索增强问答；  
如果没有 RAG，Agent 又容易因为缺失业务语义而做错决策。  
所以这两个能力在这个项目里是耦合配合的关系。

### Q42：如果面试官问你，RAG 做不好会导致什么问题，你怎么答？

**A：**

RAG 做不好一般会有两类问题：

1. **召回不到**
   - 业务术语没补齐，后续选表选字段会偏

2. **召回错了**
   - 错误知识反而会污染后续 SQL 生成和分析计划

所以 RAG 不是“加了就一定更准”，它本身也是一个需要精细设计和调优的模块。

### Q43：如果让你继续优化这个项目里的 RAG，你会先优化哪一块？

**A：**

我会优先优化两块：

1. **检索前改写**
   - 让 query rewrite 更稳定，把业务口语、上下文省略、模糊指标先改写成更适合检索的表达

2. **召回后的重排与过滤**
   - 不只是看召回到没有，还要控制召回内容是不是和当前问题强相关，避免错误知识污染后续链路

因为这个项目里的 RAG 不是一个独立问答模块，而是前置到整个分析链路里的。  
所以一旦改写质量差或者召回噪声大，后面从数据理解到 SQL 生成都会一起受影响。

### Q44：如果面试官继续追问，RAG 在你们项目里到底是怎么设计的，你怎么答？

**A：**

我会把它拆成五层来讲：

1. **知识层**
   - 知识来源不是单一文档，而是分成业务知识、Agent 私有知识这两类
2. **入库层**
   - 文档在入向量库前会先做切分、补元数据、再做 embedding
3. **检索层**
   - 先做 query rewrite，再按 agentId、知识类型等条件做检索
4. **融合层**
   - 支持纯向量检索，也支持向量 + 关键词的混合检索，再做结果融合
5. **消费层**
   - 召回结果不是只给最终答案用，而是前置注入到问题理解、数据理解、规划和 SQL 生成链路里

也就是说，这个项目里的 RAG 不是“外挂一个知识库问答”，而是一个嵌进主链路的语义增强系统。

### Q45：你们的 RAG 为什么要做知识分层，而不是所有文档都混在一起检索？

**A：**

因为不同知识的作用不一样。  
业务知识更偏指标口径、术语定义、业务规则；Agent 私有知识更偏 FAQ、文档片段、经验说明。  
如果把所有内容完全混在一起，一方面召回噪声会变大，另一方面也不利于后面做精细过滤和解释来源。

所以我更倾向于先做知识分层，再通过元数据和检索策略控制不同知识的参与方式。  
这样召回更可控，也更符合企业场景里“不同知识有不同可信度和用途”的特点。

### Q46：你们的 RAG 检索链路为什么不是用户问题直接去向量搜索，而是先做 query rewrite？

**A：**

因为用户原始提问很多时候不适合直接检索。  
它可能依赖多轮上下文，也可能有业务口语、代词、省略和模糊说法。  
如果直接拿原问题去搜，召回结果很容易不稳定。

所以我们先用大模型把问题改写成更独立、更完整、更适合检索的表达，再去做向量召回。  
这相当于先优化检索输入，再优化召回结果，通常比直接搜原问题更稳。

### Q47：你们的 RAG 为什么要做 metadata filter？

**A：**

因为这个项目不是一个公共知识库问答，而是多 Agent、多知识空间并存的系统。  
如果不做 metadata filter，很容易出现两个问题：

1. 召回到别的 Agent 的知识
2. 召回到当前虽然在库里，但业务上不应该参与本次分析的知识

所以我们会在检索时先按 agentId、知识类型，再结合数据库里的启用状态做过滤。  
本质上是在做“先缩检索边界，再算语义相似度”，这样比全量开搜更安全。

### Q48：你们为什么要支持混合检索，而不是只做向量检索？

**A：**

因为企业知识场景里，有些内容适合语义召回，有些内容更适合关键词精确匹配。  
比如术语名、缩写、特定产品名、固定指标名，这类内容有时候关键词命中比纯向量更稳。  
如果只做向量检索，可能会出现语义接近但术语不准的问题。

所以我们保留了混合检索能力，让向量检索负责语义相关性，关键词检索负责精确命中，再做融合。  
这样能同时兼顾召回率和精确率。

### Q49：那混合检索做完以后，结果怎么处理？为什么还需要融合？

**A：**

因为向量召回和关键词召回的分数体系本来就不一样，不能简单拼起来直接用。  
如果不做融合，很容易出现某一路结果系统性占优，导致另一条通道白做了。

所以我会把这一步设计成“多路召回 + 统一融合排序”。  
本质上是把不同通道的优势合并起来，而不是让它们彼此打架。  
这样整体召回质量会更稳定，也更便于后面做 TopK 控制。

### Q50：RAG 里的知识切分你们是怎么设计的？为什么不能整篇文档直接入库？

**A：**

因为整篇文档直接入库会有两个典型问题：

1. 粒度太粗，召回到的是一大段噪声，真正有用的信息被淹没
2. 文档太长，会影响 embedding 质量和后续 prompt 注入成本

所以我们会在入库前先做切分，再向量化。  
而且不是只有一种切分方式，项目里支持 token、recursive、sentence、paragraph、semantic 这些不同 splitter。  
这样可以根据知识类型选择更合适的切分粒度，而不是所有文档一刀切。

### Q51：为什么你们的 RAG 要支持多种 splitter，而不是统一一种切分策略？

**A：**

因为不同知识形态的最佳切分方式不一样。  
FAQ、问答型知识通常天然比较短，切得太碎反而会破坏语义；  
长文档、制度文档、说明手册又更适合按段落、句子甚至语义边界切分。  
如果统一只用一种 splitter，很多场景都会吃亏。

所以我们把切分策略做成可配置能力，让知识在入库阶段就尽量保留最合适的上下文粒度。  
这一步虽然看起来偏基础设施，但对召回质量影响很大。

### Q52：你们怎么控制 RAG 的错召回和噪声问题？

**A：**

我会从四层去控：

1. **检索前**
   - 先做 query rewrite，提升检索输入质量
2. **检索时**
   - 用 agentId、知识类型、启用状态做过滤
3. **检索后**
   - 用 TopK、相似度阈值、融合排序控制结果集合
4. **消费时**
   - 不是把所有召回结果无脑塞给模型，而是把真正相关的 evidence 注入后续节点

也就是说，我不会把 RAG 当成“多召回一点总没错”的系统。  
在这个项目里，错召回比少召回更危险，因为它会污染后面的 SQL 和分析计划。

## 9. Agent 系统设计类

### Q53：如果面试官直接问你，“你这个 Agent 系统是怎么设计的？”，你怎么回答？

**A：**

我会把这个 Agent 系统拆成五层来讲：

1. **接入层**
   - 接收用户问题、维护请求上下文、管理一次分析任务的 thread

2. **认知层**
   - 理解用户目标，补充业务知识，消解问题歧义

3. **规划层**
   - 根据用户目标和当前可用信息，生成分析计划，而不是直接执行

4. **执行层**
   - 按计划调用不同能力，比如 SQL 查询、Python 分析、报告生成

5. **控制层**
   - 负责重试、纠错、人工审核、中断恢复、结束条件判断

这样设计的好处是，Agent 不会退化成一个“大 prompt + 大 if/else”，而是有明确的职责分层。

### Q54：你觉得一个完整的 Agent 系统最核心应该包含哪些能力？

**A：**

我觉得至少要有六个能力：

1. 目标理解能力
2. 外部知识获取能力
3. 任务规划能力
4. 工具调用能力
5. 结果验证和纠错能力
6. 运行过程控制能力

如果缺少其中任何一个，系统就更像“问答机器人”或者“工具调用脚本”，而不是完整 Agent。

### Q55：你这个 Agent 为什么需要“规划层”，不能直接让模型一把做完吗？

**A：**

不能。  
因为复杂分析任务不是单步问题，而是多步目标达成问题。  
如果没有规划层，模型每一步都只能做局部决策，容易出现：

- 当前步骤看起来合理，但整体路径错了
- 明明该先查基础数据，却直接进入总结
- 某些问题本来需要 SQL + Python 联合处理，但系统只做了其中一步

所以规划层的本质，是把“用户问题”翻译成“可执行步骤序列”。

### Q56：Agent 的规划层和工作流编排是什么关系？两者会不会重复？

**A：**

我会把两者理解成“骨架”和“动态决策”的关系，不是替代关系。  
工作流编排负责的是系统级骨架，比如有哪些阶段、哪里能重试、哪里能审核、哪里能中断恢复。  
规划层负责的是在这个骨架里面，针对当前问题动态决定具体怎么走。  
换句话说：

- 工作流解决的是系统工程上的可控执行
- Agent 规划解决的是复杂任务里的动态决策

这个项目本质上就是“图结构骨架固定，步骤内容和部分路径动态生成”。  
所以两者不是重复，而是一个负责稳定边界，一个负责智能决策。

### Q57：你这个 Agent 是单 Agent 还是多 Agent？为什么这么设计？

**A：**

这个项目更偏**单 Agent + 多工具能力**，而不是多 Agent 协作。  
原因主要有两个：

1. 当前问题域比较聚焦，就是数据分析
2. 复杂性主要来自多阶段决策和多工具执行，不一定需要多个独立 Agent 才能解决

如果一开始就上多 Agent，会带来额外的角色划分、通信、状态同步和调试复杂度。  
所以我先把单 Agent 的规划、执行、纠错链路打透，这对当前场景更合适。

### Q58：如果面试官追问，为什么不用多 Agent 架构，你怎么答？

**A：**

我会说，不是多 Agent 一定更高级，而是要看场景是否真的需要。  
当前这个项目里，核心问题不是角色分工，而是：

- 怎么理解业务问题
- 怎么筛对数据结构
- 怎么稳定执行分析链路

这些问题在单 Agent 框架下已经可以解决。  
如果未来要扩展到“数据分析 + 指标解释 + 业务建议 + 外部系统联动”这种更复杂协作场景，我才会考虑拆成多 Agent。

### Q59：你这个 Agent 的“工具”是什么？工具系统是怎么设计的？

**A：**

我会把工具理解成 Agent 的外部执行能力。  
在这个项目里，典型工具包括：

- 查询数据库
- 执行 Python 分析
- 生成报告
- 调用知识检索能力

设计上我做的是“规划层决定用什么能力，执行层真正调用能力”。  
这样可以保证 Agent 不直接耦合某个底层实现，而是面向能力抽象来工作。

### Q60：你怎么控制 Agent 不要乱调用工具？

**A：**

我主要做三层控制：

1. **规划前控制**
   - 限定可用工具集合，不让模型随意幻想新工具

2. **执行前控制**
   - 对计划做合法性校验，检查工具名、参数和步骤结构

3. **执行中控制**
   - 对具体执行器做权限、资源和重试约束

所以工具调用不是“模型想调什么就调什么”，而是被约束在一个可控边界内。

### Q61：Agent 系统里为什么一定要有“状态管理”？

**A：**

因为 Agent 执行不是单步函数调用，而是一个持续推进的过程。  
只要是过程，就一定会涉及：

- 当前做到哪一步
- 前面拿到了什么结果
- 后面该走哪条分支
- 是否进入人工审核
- 是否需要恢复执行

这些都不是一次 prompt 能承载的，所以必须有状态管理。  
这个项目里，本质上就是用状态来驱动整个 Agent 链路推进。

### Q62：那你这个 Agent 的状态里，最关键的几类信息是什么？

**A：**

我会对外抽象成四类：

1. **输入状态**
   - 用户问题、当前上下文、任务标识

2. **认知状态**
   - 增强后的问题、知识补充结果、数据理解结果

3. **执行状态**
   - 当前计划、当前步骤、执行结果、失败原因

4. **控制状态**
   - 是否重试、是否审核、是否结束、是否恢复

这四类状态基本能覆盖一个数据分析 Agent 的完整生命周期。

### Q63：Agent 系统为什么要支持“中断与恢复”？

**A：**

因为真实业务场景里，并不是所有任务都应该一口气自动跑到底。  
有些时候需要：

- 人工审核计划
- 等待外部条件
- 用户中途修正目标

如果系统不支持中断恢复，那每次都只能从头再跑，体验和资源成本都很差。  
所以一个成熟的 Agent 系统，不只是能跑，还要能暂停、修改、继续。

### Q64：如果面试官问，“你这个 Agent 怎么保证可控性，而不是完全黑盒？”，你怎么答？

**A：**

我会从三方面回答：

1. **结构可控**
   - 链路不是完全自由生成，而是在预定义骨架内运行

2. **工具可控**
   - 工具集合、参数结构、执行权限都有限制

3. **过程可控**
   - 支持计划校验、人工审核、失败回退和结果纠错

也就是说，我不是把系统做成一个完全自由的 autonomous agent，而是一个 bounded agent，在明确边界里做动态决策。

### Q65：你怎么理解 Agent 系统里的“记忆”？

**A：**

我觉得记忆本质上是“让系统不要每一轮都从零开始”。  
它至少有两层：

1. **会话级记忆**
   - 当前任务上下文、用户前面问过什么、系统前面计划过什么

2. **知识级记忆**
   - 长期业务知识、术语、FAQ、经验规则

在这个项目里，我更强调的是“短期任务上下文 + 外部知识增强”，而不是做一个无限增长的聊天历史仓库。

### Q66：你这个 Agent 的记忆为什么没有直接做成完整聊天历史注入？

**A：**

因为完整聊天历史注入成本太高，而且噪声会越来越大。  
对于这个项目，更重要的是保留“对分析决策有价值的上下文”，而不是保留所有对话文本。  
所以我会更倾向于保留结构化的关键内容，比如：

- 用户问题
- 上一轮计划
- 关键执行结果

这是在效果、成本和稳定性之间做平衡。

### Q67：Agent 系统为什么要有“验证层”，不能只靠模型自我反思吗？

**A：**

因为模型自我反思本身还是模型输出，稳定性不够。  
如果没有独立验证层，系统很容易出现“模型自己认为自己没问题，但实际上已经错了”。  
所以我会把验证单独设计出来，比如：

- 计划结构校验
- SQL 语义校验
- 执行错误回传

这样系统不是只靠模型自己兜底，而是靠工程规则和模型能力一起兜底。

### Q68：如果让你总结一个 Agent 系统设计的核心原则，你会怎么说？

**A：**

我会总结成四句话：

1. Agent 要有目标，不只是有回答
2. Agent 要会规划，不只是会生成
3. Agent 要能调用工具，不只是会聊天
4. Agent 要可控、可验证、可恢复，不能完全黑盒

这四点也是我设计这个项目时最核心的原则。

### Q69：如果未来让你把这个 Agent 升级成平台化能力，你会怎么设计？

**A：**

我会往三个方向做平台化：

1. **能力平台化**
   - 把 SQL、Python、检索、报告这些能力做成标准工具接口

2. **流程平台化**
   - 让不同业务场景能复用 Agent 骨架，只替换配置和工具组合

3. **状态与治理平台化**
   - 统一管理记忆、审核、重试、日志、监控和权限

也就是说，从“一个项目里的 Agent”升级成“多个业务场景都能复用的 Agent 平台”。

## 五、最后建议

### 面试时你不要这样讲

- 不要一上来讲类名
- 不要一上来讲源码细节
- 不要把项目介绍成“我做了个 NL2SQL”

### 面试时你应该这样讲

1. 先讲项目目标
2. 再讲模块划分
3. 再讲你负责的核心功能
4. 再讲为什么这样设计
5. 最后讲难点、优化和不足

### 最推荐的项目开场白

> 这个项目是一个企业级数据分析 Agent，主要解决业务人员不会写 SQL、数据结构复杂、分析流程长、结果不稳定这些问题。  
> 我负责的是后端主链路和核心能力设计，重点做了数据理解、分析规划、SQL 校验纠错、Python 深度分析和执行隔离，整体把单次 NL2SQL 升级成了一条可编排、可恢复、可审核的数据分析工作流。

这段比“我做了很多节点很多接口”专业得多。
